# Proposal: App Compatibility Matrix — Format and Maintenance

**Proposed by:** Atlas (operations-manager)
**Date:** 2026-03-28
**Status:** Proposed
**Parent:** `2026-03-28-app-dev-agent.md` (Kit agent proposal)

---

## Problem

As Kit manages multiple app versions across multiple IDEA Engine versions, the question
arises: which version of an app works with which version of the Engine? Without a
machine-readable record of this, incompatible disks could be inserted into field Pis and
fail silently (or loudly) — with no technician present to diagnose.

Two places need this information:
1. **Kit** — to know which Engine version to test against when building a new app release
2. **The Engine** — to warn when an incompatible disk is inserted

---

## Proposed Format

Each app repo contains a `monitoring.yaml` file that serves dual purpose: it defines
version monitoring strategy AND records the compatibility matrix for that app.

```yaml
# app-kolibri/monitoring.yaml

app: kolibri
idea_app_version: "1.1"

monitoring:
  strategy: pex          # pex | dockerhub | custom
  pex_release_url: "https://learningequality.org/r/kolibri-pex-latest"
  cadence: monthly

services:
  - name: kolibri
    image: idea-kolibri    # IDEA-built image (from Pex)
    monitoring: pex        # monitored via pex_release_url above
  - name: db
    image: mariadb
    tag: "10.6"
    monitoring: dockerhub  # monitored via Docker Hub tags API

compatibility:
  engine_min: "0.9.0"     # minimum Engine version this app disk requires
  engine_tested: "1.0.0"  # Engine version Kit last ran tests against
  notes: "Requires Engine ≥ 0.9.0 for multi-engine disk sync support"

build:
  method: build-instance   # always — Pi-based build
  script: ./build-instance
```

For a standard Docker Hub app (e.g. Nextcloud):

```yaml
# app-nextcloud/monitoring.yaml

app: nextcloud
idea_app_version: "1.0"

monitoring:
  strategy: dockerhub
  cadence: monthly

services:
  - name: nextcloud
    image: nextcloud
    tag: "27-apache"
    monitoring: dockerhub
  - name: db
    image: mariadb
    tag: "10.6"
    monitoring: dockerhub

compatibility:
  engine_min: "0.8.0"
  engine_tested: "1.0.0"

build:
  method: build-instance
  script: ./build-instance
```

---

## How Kit Uses It

On the monthly monitoring cron run, Kit reads `monitoring.yaml` for each app and applies
the correct strategy per service:

- `dockerhub` → check Docker Hub tags API for the image; compare to current `tag`
- `pex` → fetch the release page at `pex_release_url`; parse for latest version
- `custom` → a custom script in the app repo (`scripts/check-version.sh`) handles it

When Kit builds a new release, it updates `engine_tested` to the current Engine version
and runs tests. If the test harness finds a compatibility failure, Kit opens a
`[From Kit] Feasibility` task to Axle before opening the release PR.

---

## How the Engine Uses It

The Engine reads `META.yaml` from each inserted disk. `META.yaml` is written by `build-instance`
and includes a `requires_engine_min` field populated from `monitoring.yaml`:

```yaml
# META.yaml (generated by build-instance)
diskId: "abc123"
diskName: "Kolibri v1.1"
created: 1700000000000
app: kolibri
idea_app_version: "1.1"
requires_engine_min: "0.9.0"
```

When the Engine processes a newly inserted disk, it compares `requires_engine_min` to its
own version. If the Engine is too old, it logs a warning and (in a future release) surfaces
it in the Console UI rather than attempting to start incompatible containers.

This requires a small Engine change: read `requires_engine_min` from META.yaml and compare
to the running Engine version. Axle owns this change; Kit creates a
`[From Kit] Feasibility: requires_engine_min field in META.yaml` task to trigger it.

---

## Maintenance

- Kit updates `monitoring.yaml` in each app repo whenever a new version is released
- `engine_tested` is updated on every Kit release run
- `engine_min` is updated only when a new Engine feature is required by the app (rare;
  triggered by a `[From Axle] Review` task notifying Kit of a breaking engine change)

---

## Impact

- **Kit:** owns `monitoring.yaml` in each app repo; `build-instance` reads it to populate `META.yaml`
- **Axle:** adds `requires_engine_min` check to Engine startup flow (small, one-time change;
  Kit opens a Feasibility task to trigger this)
- **Each app repo:** gains a `monitoring.yaml`
- **CEO:** no action required; compatibility warnings will appear in Console UI (future)
